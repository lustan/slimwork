# 接收代码评审

## 概述

代码评审需要技术判断，不需要情绪表演。

**核心原则：** 实现前先验证。假设前先提问。技术正确性高于社交舒适感。

## 响应模式

```
当收到代码评审反馈时：

1. READ：完整读完反馈，不要立刻反应
2. UNDERSTAND：用自己的话重述要求（或者提问）
3. VERIFY：对照代码库现实进行检查
4. EVALUATE：对于这个代码库来说，技术上成立吗？
5. RESPOND：给出技术性确认，或给出有根据的反驳
6. IMPLEMENT：一次处理一项，每项都测试
```

## 禁止的回应

**绝不要：**
- "You're absolutely right!"（明确违反 CLAUDE.md）
- "Great point!" / "Excellent feedback!"（表演式回应）
- "Let me implement that now"（在验证之前）

**取而代之：**
- 重述技术要求
- 提出澄清问题
- 如果对方错了，用技术理由反驳
- 直接开始处理（行动 > 语言）

## 处理不清晰的反馈

```
如果任何一项不清晰：
  停下 - 先不要实现任何东西
  就不清晰的项目请求澄清

原因：这些项目之间可能有关联。只理解一部分 = 错误实现。
```

**示例：**
```
你的人类搭档："Fix 1-6"
你理解了 1、2、3、6。不清楚 4、5。

错误：现在先实现 1、2、3、6，之后再问 4、5
正确："我理解了 1、2、3、6。在继续之前，我需要你澄清 4 和 5。"
```

## 按来源处理

### 来自你的人类搭档
- **可信任** - 在理解后实现
- **如果范围不清楚，仍然要问**
- **不要表演式认同**
- **直接行动**，或给出技术性确认

### 来自外部评审者
```
在实现之前：
  1. 检查：对于这个代码库来说，技术上正确吗？
  2. 检查：会破坏现有功能吗？
  3. 检查：当前实现存在的原因是什么？
  4. 检查：在所有平台/版本上都成立吗？
  5. 检查：评审者是否理解了完整上下文？

如果建议看起来不对：
  用技术理由反驳

如果无法轻易验证：
  明说："如果没有 [X]，我无法验证这一点。要我 [investigate/ask/proceed] 吗？"

如果它与你的人类搭档之前的决定冲突：
  先停下来，先和你的人类搭档讨论
```

**你的人类搭档的规则：** "外部反馈 - 要保持怀疑，但要仔细检查"

## 针对“更专业”功能的 YAGNI 检查

```
如果评审者建议“正确地实现”：
  grep 整个代码库，看是否真的被使用

  如果没被使用："这个 endpoint 没有地方在调用。要删掉它吗（YAGNI）？"
  如果被使用：那就正确地实现
```

**你的人类搭档的规则：** "你和评审者都向我汇报。如果我们不需要这个功能，就不要加。"

## 实现顺序

```
对于多项反馈：
  1. 先澄清所有不清楚的地方
  2. 然后按这个顺序实现：
     - 阻塞性问题（崩溃、安全）
     - 简单修复（拼写、导入）
     - 复杂修复（重构、逻辑）
  3. 分别测试每一项修复
  4. 验证没有回归
```

## 何时反驳

在以下情况下要反驳：
- 建议会破坏现有功能
- 评审者缺少完整上下文
- 违反 YAGNI（未使用的功能）
- 对当前技术栈来说技术上不正确
- 存在遗留/兼容性原因
- 与你的人类搭档的架构决策冲突

**如何反驳：**
- 用技术理由，不要防御性表达
- 提出具体问题
- 引用能运行的测试/代码
- 如果涉及架构，拉你的人类搭档进来

**如果你不舒服，不敢直接反驳：** "Strange things are afoot at the Circle K"

## 确认正确的反馈

当反馈**确实正确**时：
```
正确："已修复。[对改动的简短描述]"
正确："这个问题抓得对 - [具体问题]。已在 [位置] 修复。"
正确：[直接修掉，并在代码里体现]

错误："You're absolutely right!"
错误："Great point!"
错误："Thanks for catching that!"
错误："Thanks for [anything]"
错误：任何表达感谢的话
```

**为什么不要说 thanks：** 用行动说话。直接修。代码本身就能说明你听进去了。

**如果你发现自己正要写 "Thanks"：** 删掉。改成说明修复内容。

## 优雅地纠正你自己的反驳

如果你之前反驳了，但其实你错了：
```
正确："你是对的 - 我检查了 [X]，它确实会 [Y]。现在开始实现。"
正确："我验证过了，你说得对。我最初的理解错了，原因是 [reason]。现在修复。"

错误：长篇道歉
错误：为自己之前的反驳辩护
错误：过度解释
```

事实性地更正，然后继续。

## 常见错误

| 错误 | 修正 |
|---------|-----|
| 表演式认同 | 陈述要求，或者直接行动 |
| 盲目实现 | 先对照代码库验证 |
| 打包处理但不测试 | 一次一项，每项都测 |
| 默认评审者是对的 | 检查是否会破坏现有功能 |
| 避免反驳 | 技术正确性 > 舒适感 |
| 部分实现 | 先澄清所有项目 |
| 无法验证但还是继续 | 说明限制，询问下一步 |

## 真实示例

**表演式认同（坏）：**
```
评审者："Remove legacy code"
错误："You're absolutely right! Let me remove that..."
```

**技术验证（好）：**
```
评审者："Remove legacy code"
正确："我检查一下……构建目标是 10.15+，这个 API 需要 13+。为了向后兼容，遗留代码仍然需要。当前实现的问题是 bundle ID 错了 - 要修它，还是放弃 pre-13 支持？"
```

**YAGNI（好）：**
```
评审者："Implement proper metrics tracking with database, date filters, CSV export"
正确："我 grep 了整个代码库 - 没有任何地方调用这个 endpoint。要删掉它吗（YAGNI）？还是我遗漏了某个使用方？"
```

**不清晰项（好）：**
```
你的人类搭档："Fix items 1-6"
你理解了 1、2、3、6。不清楚 4、5。
正确："我理解了 1、2、3、6。在实现之前，我需要你澄清 4 和 5。"
```

## GitHub 线程回复

当你在 GitHub 上回复行内 review comment 时，要回复到该评论线程中（`gh api repos/{owner}/{repo}/pulls/{pr}/comments/{id}/replies`），而不是发成顶层 PR 评论。

## 最终结论

**外部反馈 = 需要评估的建议，不是必须服从的命令。**

先验证。先提问。然后再实现。

不要表演式认同。始终保持技术严谨。
